# Политика безопасности

## Поддерживаемые версии

| Версия | Поддержка |
|--------|-----------|
| `master` (текущая разработка) | ✅ |

MKG — MVP/хакатонный проект; критические исправления по возможности вливаются в `master`.

## Сообщить об уязвимости

**Не создавайте публичный issue** с описанием уязвимости до её устранения.

### Как сообщить

1. **Предпочтительно:** [GitHub Security Advisory](https://github.com/Pesimistixxx/NrnklKlubok/security/advisories/new) (*Report a vulnerability*).
2. **Альтернатива:** приватное сообщение maintainer'у через GitHub (**Pesimistixxx**) или issue с минимумом деталей и просьбой связаться лично.

### Что указать

- тип проблемы (инъекция, утечка секретов, SSRF, RCE и т.д.);
- затронутый компонент (gateway, worker, agents, Neo4j/Qdrant);
- шаги воспроизведения;
- возможное влияние;
- предложение исправления (если есть).

### Чего ожидать

- подтверждение в течение **7 рабочих дней** (best effort);
- оценка и план исправления;
- координация публикации после патча.

## Рекомендации для развёртывания

- Не коммитьте `.env` с реальными ключами Yandex Cloud.
- Меняйте пароли Neo4j/Postgres по умолчанию в production.
- Не выставляйте gateway и Neo4j Browser в интернет без аутентификации и TLS.

## RBAC и доступ в MKG (MVP)

В hackathon MVP **нет криптографической аутентификации**: роль — самодекларация клиента (`X-MKG-Role`), API на localhost открыт. При этом **фильтрация по грифу (`classification`) на чтении** уже применяется (документы, поиск, граф, RAG) на основе матрицы role × classification. Ролевая модель, L5 `SecurityRole`, аудит и roadmap production-auth — в [`Docs/28_access_and_security.md`](Docs/28_access_and_security.md) (в приложении: **Справка → Доступ и безопасность**).
